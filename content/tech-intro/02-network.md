打开浏览器访问一个网页，背后发生的事情可以拆成三个基本问题：怎么找到对方的机器、找到机器上的哪个程序、传输中怎么保证数据不丢不乱。IP、端口、DNS、TCP、HTTP 分别负责其中一块，合在一起构成日常使用的互联网通信。

## 两台机器要通信，要解决哪些问题

两台机器相距几千公里，中间经过无数路由器和交换机。数据从出发到抵达，至少要回答三件事：

第一，数据要送到哪台机器？网络上的机器数以亿计，必须有一套地址系统能唯一标识每台机器的位置。

第二，到了那台机器之后，送给哪个程序？一台服务器上可能同时跑着数十个程序，数据不能随机派发。

第三，中间的传输可靠吗？网络本身不会保证数据一定送达，路由器可能丢包，数据包可能乱序抵达。

接下来每一节处理其中一块。

## IP 与端口：找到机器，找到机器上的程序

IP（Internet Protocol，互联网协议，负责在网络中寻址和路由数据包的基础协议）地址是网络层的地址，用于定位一台具体的机器。最常见的 IPv4 地址是四段数字，比如 `192.168.1.1`，每段 0 到 255。每个联网设备都有一个 IP 地址，路由器在转发数据包时就靠这个地址决定下一步往哪走。

IP 地址解决了"找到哪台机器"的问题，但还没有解决"找到机器上哪个程序"的问题。这里引入端口（Port）。端口是一个 0 到 65535 的数字，操作系统用它来区分同一台机器上不同的网络程序。一个程序要接收网络数据，必须先绑定一个端口，相当于在机器上挂了一块门牌号。

常见的端口号有约定俗成的用途：80 对应 HTTP，443 对应 HTTPS，22 对应 SSH。这些端口号不是强制规定，只是行业惯例，服务器完全可以把 HTTP 服务跑在别的端口上。

一条网络连接由四个要素唯一确定：源 IP、源端口、目标 IP、目标端口。有了这四个要素，操作系统就知道该把收到的数据包交给哪个程序处理。

## DNS：把域名翻译成 IP

人记不住 IP 地址，所以引入了域名（Domain Name）。`github.com`、`google.com` 这类域名背后都对应一个或多个 IP 地址。把域名翻译成 IP 地址这件事，由 DNS（Domain Name System，域名系统，一套把人类可读的域名转换为机器可用的 IP 地址的分布式查询系统）负责。

DNS 查询的过程大致如下。浏览器先检查自己的缓存里有没有这个域名对应的 IP；没有就问操作系统的缓存；再没有就问本地 DNS 服务器（通常是网络运营商提供的）。如果本地 DNS 服务器也没有缓存，它会代替你去问根 DNS 服务器，根服务器告知应该去问哪个顶级域名服务器（比如负责 `.com` 的），顶级域名服务器再指向权威 DNS 服务器，最终拿到实际的 IP 地址，返回给浏览器，并在各级缓存里保存一段时间。

整个查询过程通常在几十毫秒内完成，日常感知不到。缓存的存在让大部分查询在前几步就能拿到结果，不必每次都走完整链条。DNS 解析完成后，浏览器拿到目标 IP，才能开始建立网络连接。

## HTTP 与 HTTPS：浏览器和服务器在说什么

有了 IP 和端口，浏览器知道数据往哪里发。但发什么、怎么发，还需要一套规则，这就是 HTTP（HyperText Transfer Protocol，超文本传输协议，规定了浏览器和服务器之间如何发送和接收数据的应用层协议）。

HTTP 是一种请求-响应协议：浏览器（客户端）主动发起一个请求，服务器处理后返回一个响应。请求包含三部分：请求行（说明要做什么，比如 `GET /index.html`）、请求头（附带的元信息，比如浏览器类型、可接受的内容格式）、请求体（POST 请求时携带的数据）。响应同样包含三部分：状态行（一个三位数字状态码，200 表示成功，404 表示找不到，500 表示服务器出错）、响应头、响应体（实际返回的内容，可以是 HTML、图片、JSON 等）。

HTTP 协议本身是明文传输的，中间任何一个节点都能看到传输的内容。HTTPS（HTTP over TLS，在 HTTP 基础上加入了 TLS 加密层，让传输内容对中间节点不可见）在传输层和应用层之间增加了一层 TLS（Transport Layer Security，传输层安全协议，负责对通信内容加密和验证服务器身份），服务器通过数字证书向浏览器证明自己的身份，双方协商出一个对称密钥，后续所有内容都用这个密钥加密。

HTTPS 默认走 443 端口，HTTP 默认走 80 端口。现代浏览器会对仍在使用 HTTP 的网站显示"不安全"警告，主流网站已全面切换到 HTTPS。TLS 握手的具体字段和步骤有专门的规范，理解到"加密传输、证书验证身份"这一层就够用了。

## TCP 与 UDP：保证送达 vs 尽力而为

HTTP 和 HTTPS 定义了应用层的通信格式，但数据在网络中真正传输依赖传输层协议。最主要的两个是 TCP 和 UDP。

TCP（Transmission Control Protocol，传输控制协议，一种保证数据按顺序、完整送达的传输层协议）在发送数据前先和对方建立连接，传输过程中对每个数据包进行编号，接收方收到后返回确认，发送方没收到确认就会重传。丢包、乱序、重复——TCP 在传输层统统处理掉，上层应用拿到的是一个干净的字节流。代价是有连接建立的开销，以及因为等待重传产生的延迟。HTTP、HTTPS 都跑在 TCP 上。

UDP（User Datagram Protocol，用户数据报协议，一种不建立连接、不保证送达的传输层协议）则直接发包，不等对方确认，丢了就丢了，不重传。好处是延迟极低，适合对实时性要求高、偶尔丢一两个包也能接受的场景，比如视频通话、在线游戏、DNS 查询。如果应用需要可靠性，可以在应用层自己实现，比如 HTTP/3 底层的 QUIC 协议就是这样做的。日常理解记到"TCP 保证送达、UDP 不保证"这一层就够了。

## URL 到响应：完整链路

把上面几块串起来看一次完整的请求。在浏览器地址栏输入 `https://example.com/page`，按下回车：

浏览器解析 URL，提取出域名 `example.com`，发起 DNS 查询，拿到对应的 IP 地址。

浏览器向这个 IP 的 443 端口发起 TCP 连接，连接建立后，因为是 HTTPS，还要进行 TLS 握手，完成加密层的协商和服务器身份验证。

握手完成，浏览器构造一个 HTTP GET 请求，请求路径是 `/page`，通过加密的 TLS 通道发给服务器。

服务器接收到请求，查找对应的资源，构造 HTTP 响应（状态码 200，响应体是 HTML 内容），同样通过 TLS 加密后发回。

浏览器收到响应，解析 HTML，渲染成页面。

IP 负责找到服务器，端口负责找到具体程序，DNS 把域名翻译成 IP，TCP 保证数据完整有序，HTTP 定义了请求和响应的格式，TLS 在 HTTPS 中保证内容不被看到。每一层各司其职，组合在一起，就是每天使用的互联网通信。
