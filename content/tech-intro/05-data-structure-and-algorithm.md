程序在运行时，数据必须以某种形式存放在内存里。放的方式不同，能做的操作和速度都会不同。数据结构（Data Structure）就是描述"怎么组织数据"的概念，算法则是"处理这些数据的步骤"。两件事合在一起，决定了一段程序好不好。

## 数据结构是什么：组织方式决定能力

同样是一批用户的购买记录，存成一个按时间排列的序列，和存成一张以用户 ID 为键的映射表，能做的操作截然不同。前者适合按时间顺序遍历，后者能在常数时间内找到某个用户的所有记录。数据结构的选择不是技术细节，而是在最早期就决定了系统能做什么、在什么规模下还能保持速度。

## 数组与链表：连续与跳转的取舍

数组（Array）的本质是一块连续的内存区域。因为每个元素占的空间相同、起始地址已知，所以访问任意位置的元素只需做一次地址计算，速度是常数级——不管数组有一百个元素还是一百万个，取第 N 个的代价相同。视频播放器的帧缓冲区、图像处理里的像素矩阵，都是数组的典型场景：需要按下标快速读写，数据量固定或变动不频繁。代价是中间插入和删除——在第 3 个位置插入一个新元素，后面所有元素必须向后移动一位，数据量越大，这个操作越慢；动态大小的数组（比如 Go 的切片、C++ 的 vector）在容量不够时还要重新申请内存并复制全部数据，扩容本身也有额外开销。

链表（Linked List）解决了这个问题。每个节点存放数据和"下一个节点在哪里"的指针，节点散落在内存各处，彼此通过指针相连。在已知位置插入或删除一个节点，只需修改相邻节点的指针，代价是常数级。消息队列的底层实现、操作系统的进程调度列表，都用到链表——频繁地在头部或中间增删元素，链表有明显优势。

但链表放弃了随机访问。要找第 N 个节点，必须从头开始逐个跳转，节点越多越慢。另一个代价是内存局部性差：数组在内存里连续，CPU 缓存读取一片连续内存的效率很高；链表节点分散，缓存命中率低，遍历时实际速度比理论估算要慢一些。

两者之间没有绝对的优劣，而是取决于场景里读还是写更频繁、随机访问还是顺序遍历为主。

## 哈希表：为什么能"瞬间"找到一个东西

从一万个用户里按用户名查某人，如果用数组顺序存储，最坏情况要从头扫描到尾。哈希表（Hash Table）把这个问题解决了——无论里面存了多少条数据，查找一个 key 的时间平均是常数级。

原理是哈希函数（Hash Function）：把 key 通过一个计算变成一个数组下标，数据就存在那个位置。查找时同样对 key 做相同计算，直接跳到对应位置取值，不需要遍历。这是"瞬间找到"的实质：不是真的瞬间，而是把查找变成了一次计算加一次直接访问。

现实工程中，哈希表无处不在。HTTP 框架在启动时把路由路径（如 `/api/users`）映射成处理函数，存进哈希表，请求到来时直接查表找到对应处理器；编译器用哈希表记录变量名和它的类型、作用域；数据库的查询缓存也是 key-value 哈希表结构。上一篇提到的 Redis，其底层核心数据结构就是哈希表。

哈希表有一个需要处理的问题：不同的 key 经哈希函数计算后可能落在同一个位置，称为哈希冲突（Collision）。常见的处理方式是拉链法——每个位置维护一个链表，冲突的元素都挂在上面。冲突多了，链表变长，查找退化成顺序扫描，效率下降。好的哈希函数会让 key 均匀分布，避免冲突集中；当元素数量与桶数量的比例（负载因子）超过阈值时，哈希表会自动扩容并重新分配所有元素。

## 树与图：现实世界的层级和关系

很多数据天然是层级结构或网状结构，数组和链表都不适合表达这类关系。树（Tree）和图（Graph）就是为此而生的数据结构。

树是有父子层级关系的结构，每个节点有零到多个子节点，且节点之间没有环。文件系统的目录、HTML 的 DOM 嵌套、公司组织架构，都是典型的层级结构——树能自然表达父子关系和从根到叶的遍历方向。对树的遍历可以从根往下逐层展开，也可以从叶子往上汇总——这两种方向对应不同的操作需求。

树在查找场景有专门的变体（如二叉搜索树），每次比较能排除一半范围，效率远高于线性扫描。数据库索引底层用的也是树形结构（第 03 篇已提到），原理一致。

图（Graph）比树更通用：节点之间可以有任意方向的边，可以有环，没有严格的父子层级。社交网络里用户之间的关注关系是图；地图导航里城市和道路是图；微服务架构里服务之间的调用依赖关系也构成一张图。图的常见操作是找到节点间的连通关系或最短路径，导航软件和微服务依赖分析都依赖这一结构。

## 算法与复杂度：同一件事可以快也可以慢

数据结构决定了数据怎么组织，算法（Algorithm）决定了怎么处理这些数据。同一个问题可以有多种算法，效率差别可能极大。从一个一百万条记录的有序列表里找某个值，逐条扫描最坏需要一百万次比较；二分查找每次排除一半，最多二十次就能找到。输入规模一样，两种算法的速度差了几万倍。

衡量算法快慢的工具是**复杂度**（Complexity），通常用大 O 记法（Big O Notation）描述，把无关常数省略掉，只关注规模增长带来的代价变化。O(n) 表示代价随数据量线性增长，O(log n) 每次能排除一半所以增长极慢，O(n²) 则是嵌套扫描——数据量翻倍时代价涨四倍。复杂度描述的是"规模增大时代价怎么变"，而不是具体运行多少秒。

复杂度的重要性在高并发场景下特别突出。单次请求的差距是微秒级，但一台服务器每秒处理十万请求时，O(n) 和 O(n²) 的差距可能是服务能否撑住流量的分水岭。工程里选择算法，不是追求理论上最优，而是在数据规模和使用场景下找到足够快的解法。

选对数据结构、选对算法，是让程序在数据量增大时仍然可用的根本手段——这两件事不是竞赛题目里的考点，而是工程代码质量的基础。
