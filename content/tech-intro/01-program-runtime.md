打开电脑，双击一个应用，它就跑起来了。中间涉及的几个关键环节——编译、加载、进程创建、内存分配、CPU 调度——正是理解"进程崩了""内存泄漏""CPU 跑满"这些概念的基础。

## 从源码到可执行文件

程序员写的代码是人可以读的文字，比如 `for i := 0; i < 10; i++` 这样的一行。计算机的 CPU（Central Processing Unit，中央处理器，是计算机执行所有计算指令的核心芯片）只认识自己的机器指令，是一串二进制数字，跟人类语言相差甚远。从"人写的代码"到"CPU 能执行的指令"，需要经过翻译。

翻译方式主要有两种。

第一种是**编译**（Compile），指把整个源代码文件一次性翻译成机器指令，产出一个独立的可执行文件。C、Go、Rust 都走这条路。编译完成后，这个可执行文件就不再依赖原始源码，可以直接运行。

第二种是**解释执行**（Interpret），指在运行时逐行读取代码并即时翻译成指令执行，不产生独立的可执行文件。Python 是典型的解释型语言。此外还有 Java 这类"先编译成中间格式（字节码），运行时再由虚拟机解释执行"的混合方式。对日常理解程序运行来说，知道"整体翻译"和"逐行翻译"两种模式的存在即可，不必追究每种语言的具体归类。

编译完成后还需要**链接**（Linking）——把程序依赖的外部库（别人写好、可复用的代码集合）并入程序，形成完整的可执行文件。知道"可执行文件里不只有你自己写的代码"就够了。

## 程序加载进内存，成为进程

磁盘上的可执行文件是静止的数据，跑起来之前什么也不会发生。要让程序动起来，操作系统（Operating System，管理计算机硬件资源、为应用程序提供运行环境的基础软件，例如 Windows、macOS、Linux）需要把它加载到内存中，创建一个**进程**（Process，操作系统视角下一个正在运行的程序实例，拥有独立的内存空间和系统资源）。

一个可执行文件被启动多次，就是多个独立的进程，它们互不干扰——一个进程崩溃不会直接导致另一个进程崩溃（除非它们之间有通信关系）。

进程在内存里有自己的地址空间，操作系统给每个进程划出一块"隔离区域"，让它们以为自己独占内存。这是通过虚拟内存机制实现的，不展开细节，知道每个进程有自己的内存视图就够了。

## 进程与线程：一个程序怎么同时做多件事

进程是操作系统管理资源的最小单位，但一个进程内部往往需要同时做多件事。比如一个音乐播放器，要同时播放声音、响应用户点击、检测网络状态。用多个进程来处理这些任务代价太高，于是引入了**线程**（Thread，进程内部的一个执行单元，多个线程共享同一个进程的内存空间，但拥有各自独立的执行流程）。

一个进程可以包含多个线程，这些线程共享进程的内存，可以方便地读写同一份数据。代价是共享带来的风险——多个线程同时修改同一块数据时，如果处理不当就会出错，这就是程序员常说的**线程安全**问题（指多个线程并发操作共享数据时，程序能产生正确结果）。

## 内存：程序的工作空间

内存（Memory，计算机用来临时存放程序运行期间所需数据和指令的地方，断电即清空，速度远比磁盘快）是程序运行期间的工作空间。每个进程在内存里的空间大致分为几个区域，其中与日常编程最相关的是**栈**（Stack）和**堆**（Heap）。

**栈**（Stack，一块自动管理的内存区域，用于存放函数调用时的局部变量和调用关系）是函数调用时自动申请、函数返回时自动释放的。程序员通常不需要手动管理。栈的空间较小，如果函数嵌套调用太深（比如递归没有终止条件），就会出现"栈溢出"（Stack Overflow）错误。

**堆**（Heap，一块由程序主动申请和释放的内存区域，用于存放生命周期不确定的大块数据）需要程序员自己管理申请和释放。忘了释放就会出现**内存泄漏**（Memory Leak，程序不断申请内存但不释放，导致可用内存越来越少直到耗尽）。现代语言如 Go、Java、Python 都有垃圾回收机制（Garbage Collection，自动检测并回收不再使用内存的机制），代替程序员处理这件事。

## CPU 怎么"同时"跑多个程序

一台普通电脑同时开着浏览器、音乐软件、文本编辑器，感觉上三件事同时在跑。但 CPU 实际上能同时执行的任务数量取决于它的**核心数**（Core，CPU 内部独立的处理单元，多核 CPU 即含有多个 Core）。一个单核 CPU 同一时刻只能执行一条指令流。

"多任务"的实现方式是**时间片轮转**——操作系统把 CPU 时间切成极小的片段（通常是几毫秒），依次分配给不同的进程或线程。每个进程执行一小段时间，然后被暂停，换下一个进程执行，轮流往复。由于切换速度极快，人感觉不到这种切换，认为是"同时在跑"。这种切换过程叫做**上下文切换**（Context Switch，操作系统保存当前进程的运行状态，加载另一个进程的运行状态的过程）。

上下文切换有开销，频繁切换会消耗额外的 CPU 时间。这也是为什么线程或进程数量不是越多越好——开太多任务，切换的成本本身就会把 CPU 资源消耗掉一大块。

多核 CPU 的出现让真正的并行成为可能：4 个核心可以同时执行 4 个任务，而不是靠时间片错开。但操作系统的调度（决定哪个任务在哪个核心上运行）仍然非常复杂。

## 为什么需要操作系统居中协调

如果没有操作系统，每个程序都要自己直接和 CPU、内存、磁盘打交道：两个程序同时写磁盘同一区域谁说了算？一个程序占完所有内存其他程序怎么办？某个程序死循环占着 CPU，其他程序永远得不到执行机会。

操作系统的核心职责是居中协调：

- **资源分配**：内存、CPU 时间、磁盘 I/O（Input/Output，程序向存储或外设读写数据的操作）等有限资源由操作系统统一分配，防止某个进程独占。
- **隔离与保护**：每个进程在自己的内存地址空间内运行，不能随意读写其他进程的内存，防止一个程序的错误蔓延到整个系统。
- **抽象接口**：程序员不需要直接操控磁盘的物理扇区，而是用"打开文件""写文件"这类标准接口，叫做**系统调用**（System Call，应用程序向操作系统请求服务的标准接口）。

这套机制让现代操作系统可以同时运行几十上百个进程而不互相踩踏。

这条链路从头串一遍：源码经过编译或解释成机器指令，可执行文件被操作系统加载进内存成为进程，进程里的线程等待被分配 CPU 时间，CPU 拿到指令后在自己的核上执行，需要数据就从内存取，需要持久化就调用系统调用写磁盘。这便是一个程序跑起来时，在看不见的地方发生的事。
