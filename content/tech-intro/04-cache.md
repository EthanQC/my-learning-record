数据库能稳定地存数据、可靠地查数据，但它的速度取决于磁盘的读写速度。大部分业务场景里，同一份数据会被反复查询——用户信息、商品详情、配置项——每次都走磁盘代价太高。缓存（Cache，把频繁访问的数据临时存放在更快的存储介质里、下次直接读取的机制）就是把这些高频数据提前搬到内存里，让多数请求不必再触碰磁盘。

## 内存与磁盘的速度差距

内存和磁盘在速度上不是同一个量级。普通机械磁盘的随机读取延迟在毫秒级（数十毫秒），即使是固态硬盘（SSD）也在百微秒级；内存的随机读取延迟在纳秒到几十纳秒之间，两者相差几个数量级。

这个差距在单次查询时感知不明显，但在高并发场景下会被迅速放大。一台服务器每秒收到十万次请求，如果每次都查磁盘，磁盘 I/O 很快就成为瓶颈；把结果存入内存后，相同压力下系统可以轻松应对。这就是缓存存在的根本原因：不是磁盘存储"不好"，而是访问速度上有结构性差距，把热数据放内存可以把这个差距消除掉。

## Redis 是什么，为什么常被用作缓存

Redis（Remote Dictionary Service）是一个开源的内存数据存储系统，数据主要保存在内存里，是目前工程中最常见的缓存层选择。

Redis 常被用来做缓存，首先是性能上的优势：数据保存在内存里，读写延迟极低，单实例每秒能处理十万级别的请求（QPS），足以把磁盘数据库承受的查询压力大幅吸收掉。

工程上同样便利：Redis 支持字符串、列表、集合、有序集合等多种数据类型，能直接存储业务中常见的结构，不必在应用层做额外转换；过期时间（TTL）也内置在存储模型里，给一条数据设置多少秒后自动失效，不需要另写清理逻辑。加上单线程命令执行模型，简单操作天然具备原子性，并发写入时不会产生数据混乱。

Redis 也能承担消息传递、分布式锁等其他职责，但缓存是它最典型的使用场景。实际工程里，通常是关系型数据库负责存储和持久化，Redis 架在前面做缓存层，拦截掉绝大多数查询流量，只有 Redis 找不到的数据才去查数据库。

## 缓存命中与未命中

系统优先从 Redis 读数据，读到即返回，称为**缓存命中**（Cache Hit）；读不到才回退到数据库查询，并把结果写回 Redis 供下次使用，称为**缓存未命中**（Cache Miss）。

命中率（命中次数占全部请求的比例）是衡量缓存效果的核心指标。命中率高意味着大多数请求被 Redis 拦下，数据库压力低；命中率低意味着大量请求仍然穿透到数据库，缓存形同虚设。

影响命中率的因素主要有两个：一是缓存了哪些数据，要被频繁查询的数据才有缓存价值；二是缓存里的数据是否还"有效"，过期或被淘汰的数据无法命中。

## 过期与淘汰：缓存放不下了怎么办

Redis 给每条数据提供了 TTL（Time To Live，存活时间，数据在缓存里最多保留多久）机制。设置了 TTL 的数据，到期后自动失效，下次访问时视同未命中。TTL 有两个用途：一是让数据定期从缓存中消失，避免长期占用内存；二是控制缓存与数据库之间的数据不一致窗口——TTL 越短，缓存里的旧数据消散越快，不一致的时间越短。

但光靠 TTL 不够，Redis 的内存是有上限的。当占用内存接近设定上限时，会触发**内存淘汰**（Eviction）机制——主动删除一部分数据来腾出空间。

Redis 提供了多种淘汰策略，覆盖不同的优先级取向：优先淘汰最近最少被访问的数据（LRU，最近最少使用），把"冷数据"先清掉；优先淘汰访问频次最低的数据（LFU，最低频使用），保留真正高频的数据；或者配置成内存满了就直接拒绝写入，适合不能丢缓存数据的特殊场景。

实际工程里，不会对所有数据都设置相同的 TTL，也不会寄望于某一个淘汰策略解决所有问题——这两个机制配合使用，让缓存空间在有限内存里尽量装最有价值的数据。

## 缓存不是免费的：一致性问题

把数据同时存在数据库和 Redis 两个地方，就有了**一致性**（Consistency，两处数据是否始终保持相同状态）问题：数据库里的数据被修改之后，Redis 里的旧数据应该怎么处理？

最常见的处理方式叫**旁路缓存**（Cache Aside）：写操作时，先更新数据库，再删除 Redis 里对应的缓存条目；读操作时，先查 Redis，未命中再查数据库并把结果写回 Redis。删除而不是更新缓存的原因在于：如果改完数据库再去更新 Redis，两步操作之间可能有其他请求并发写入，导致 Redis 里存入了一个更旧的值，反而加剧不一致。先删缓存、等下次读请求触发回填，比直接覆盖更安全。

即便如此，这种方案在极端并发下仍然存在短暂的不一致窗口。一个典型的问题场景是**缓存击穿**（Cache Breakdown，热点数据的缓存恰好在高并发时失效，大量请求同时穿透到数据库）：热点 key 过期的瞬间，所有并发请求同时发现缓存未命中，一起涌向数据库，瞬间把数据库打垮。解决方法是用互斥锁把"查数据库并写回缓存"这个步骤串行化——只让一个请求去查，其余请求等它写回后再读缓存。

还有一类问题叫**缓存穿透**（Cache Penetration，请求的数据在数据库里根本不存在，每次都穿透缓存直击数据库）：通常发生在有人构造大量不存在 ID 的请求时，因为数据库没有，Redis 也不会缓存，每次都落到数据库。常见对策是把"不存在"这个结果本身也缓存起来，设置短 TTL，让后续相同请求命中"空值缓存"而不再查数据库。

这些一致性问题没有零代价的解法：TTL 越短、一致性越好，但缓存命中率越低；强制同步更新缓存、一致性越好，但写操作的复杂性和延迟越高。工程上的选择总是在"多快能看到最新数据"和"缓存发挥了多少效果"之间权衡，而不存在两全其美的配置。
