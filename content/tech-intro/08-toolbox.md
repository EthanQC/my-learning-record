协作、部署、稳定运行，离不开一套工具基础设施——Git 管版本、Linux 命令行操控服务器、Docker 解决环境差异、CI/CD 让发布自动化，这四样几乎每个开发者每天都会碰到。

## Git：代码变更的历史记录与协作

没有版本控制时，改动代码最常见的应对方式是手动复制——保存一份"昨天能跑的版本"、一份"今天改了支付的版本"、一份"备份-最终-真的最终"。这在单人小项目里勉强能撑，但一旦多人协作，麻烦立刻出现：两个人同时修改了同一个文件，谁的版本算数？某个改动引入了 bug，能回到出问题之前那一刻的状态吗？

Git（一个开源的分布式版本控制系统，由 Linus Torvalds 在 2005 年为管理 Linux 内核代码而创建）把代码的每次改动作为一次**提交**（commit）记录下来，每次提交保存当时整个项目的状态快照以及变更描述。改错了可以回到任意历史提交，不需要手动维护备份副本。

Git 的另一个核心抽象是**分支**（branch）。分支让多个改动可以同时并行推进，互不干扰。主干代码保持稳定，新功能或 bug 修复在独立分支上开发，开发完成再合并回主干。两个人在各自分支上改同一文件，Git 会在合并时提示哪些地方有冲突，让开发者手动确认如何取舍，而不是静默地覆盖对方的工作。

Git 是本地运行的工具，代码历史保存在本地仓库里。GitHub 则是建立在 Git 之上的线上托管平台，把本地仓库同步到云端，供团队成员共同访问。两者是不同层面的东西——Git 是版本控制工具，GitHub 是托管和协作服务。

## Linux 命令行：操控服务器的基本方式

绝大多数后端服务跑在 Linux 服务器上，而这些服务器通常没有图形界面——没有桌面、没有鼠标，只有一个文本终端。命令行就是操控这类机器的基本方式。对后端工程师来说，从登录服务器排查日志，到执行批量操作，命令行几乎是日常工作的起点。

图形界面的本质是把底层操作包装成按钮和菜单，用起来直观，但在服务器场景里有明显局限。图形界面消耗资源：跑一套完整的桌面环境需要额外的 CPU 和内存，服务器上的资源应该给业务程序用。更重要的是它难以自动化：人工点击无法写成脚本，批量操作时效率极低。命令行操作天然可以串联成脚本、批量执行、作为其他工具的输入，这是它在服务器场景里难以替代的原因。

Linux 命令行的常用动作大致分几类。文件操作是最基础的，浏览目录、读写文件、移动复制、调整权限。进程管理则是查看当前运行着哪些进程、某个服务占用了多少 CPU 和内存、让某个进程结束——排查服务器问题时这类操作是第一步。网络诊断是确认服务是否在监听正确端口、两台机器是否能互通。

命令不需要全部记住，形成"命令行可以做这件事"的直觉比记住命令名更重要。

## 容器与 Docker：把程序连同环境一起打包

"在我机器上能跑，到服务器上就报错"是开发中极其常见的抱怨。最常见的来源是依赖版本：本地用的某个库是 2.1，服务器上是 2.0，行为细节不同。操作系统差异、环境变量缺失，也是同一类问题。

Docker（目前最主流的容器化平台）给出的解决思路是把程序和它依赖的所有环境打包在一起，一起交付，引入了**容器**（Container）的概念。容器是一个独立的运行单元，里面包含程序本身、运行时环境、系统库、配置——所有运行一个程序所需的东西。容器在任何支持 Docker 的机器上运行，得到的是完全相同的环境，与宿主机器的环境无关。

容器和虚拟机（Virtual Machine）都能提供隔离的运行环境，但出发点不同。虚拟机模拟完整的硬件，在虚拟硬件上安装完整的操作系统，隔离程度高但启动慢、占用资源多。容器共享宿主机的操作系统内核，只隔离文件系统、进程和网络，因此启动时间以秒计、资源开销远小于虚拟机，更适合频繁创建和销毁的场景。

Docker 镜像（Image）是容器的模板，描述了容器里装什么。镜像可以发布到镜像仓库，任何人拉取同一个镜像，运行出来的容器都是相同的环境。这让"把程序交给别人运行"变成了"把镜像给别人，他们直接 run"，不再需要附赠一份几十步的环境配置文档。

## CI/CD：持续集成与持续交付

没有自动化流程时，代码从写完到上线需要人工逐步介入：通知测试、触发构建、核对结果、手动部署，每个环节都可能遗漏，流程也因人而异。

CI/CD（持续集成与持续交付/部署）让代码变更能快速、自动地经过验证并推向生产。CI（持续集成）的核心是：每次代码提交到仓库，自动触发编译、测试流程，问题在进入主干之前就暴露出来，测试不通过则提交立即被标记失败。

具体来说，CI 的流程会拉取最新代码、运行编译、执行测试、生成报告，这套流程通常在几分钟内完成。

CD（持续交付或持续部署）在 CI 的基础上继续往前：测试通过后，自动把新版本部署到目标环境。持续交付是自动构建出可部署的产物，在人工确认后发布；持续部署则完全自动，测试通过即部署，不需要人工触发。两者的差别在于最后一步是否需要人批准。

这套机制最直接的价值是让发布从"大版本攒一批改动一起上"变成"改动随时可以上"。每次发布的改动量越小，出问题时排查范围越小，回滚代价也越小。自动化的测试保证每次上线前都经过同样的验证标准，而不是依赖某个人有没有记得手动跑一遍。工程实践越来越把 CI/CD 视为标准配置，而不是可选的额外投入。
