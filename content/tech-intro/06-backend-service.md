用户操作触发的那一半逻辑，发生在服务器上看不见的地方——对外是一组 API，内部是处理请求的程序，背后是存数据的数据库。这一半就是后端服务。

## 后端服务做什么

前端（网页、App）负责用户能看到的界面和交互，后端负责数据存储、业务规则、权限控制。前端能展示的数据，几乎全部来自后端；前端触发的每一个"提交"操作，也几乎全部由后端来执行和判断结果。

后端服务通常承担鉴权、读写数据、执行业务规则、对接其他服务等职责。规模小时一台机器够用，规模增大就需要扩展和冗余。

## API：服务对外提供能力的接口

后端程序不是直接把源码交给前端去运行，而是对外暴露一组**接口**，前端通过这些接口发请求、拿结果。这组接口就叫 API（Application Programming Interface，应用程序编程接口，服务提供给外部调用者的一组约定好的请求与响应规则）。

API 规定了几件事：调用方发什么样的请求（请求的路径、携带哪些参数、用什么格式）、服务会返回什么样的结果（数据结构、成功还是失败的标识）。调用方不需要关心后端代码怎么写的，只要按照约定发请求，就能拿到预期的数据。

比如"获取用户信息"可能是向 `/api/users/123` 发一个请求，服务返回包含用户名称、头像等字段的 JSON 数据——这是一个典型的约定。

API 不只是前端在调用。两个后端服务之间也通过 API 互相调用——订单服务下单时调用库存服务扣减库存，库存服务调用通知服务发送提醒。服务间通信基本都依赖 API——调用方只要遵守约定，服务内部实现可以随时演进，互不影响，这也是整个系统能够拆开分工的基础。

## 服务器：硬件还是软件

物理意义上的服务器是一台安装在机房里的计算机，配置比家用电脑高、长期稳定运行、通过网络对外提供服务。日常沟通里"服务器"指向这两类中的一种，不区分会造成混乱。它本身只是硬件——有 CPU、内存、磁盘、网卡，跟普通电脑没有本质区别，只是专为长时间稳定运行和高负载场景设计。

软件意义上的服务器则是跑在这台机器上的程序——HTTP 服务器、数据库服务器、缓存服务器，这些都是程序，在某台机器上监听特定的端口，接受来自网络的连接和请求，处理后返回结果。一台物理机器上可以同时运行多个这样的程序，每个监听不同的端口。

工程讨论里说"服务器挂了"，多数情况下指的是程序进程崩溃或异常退出，而不是机器断电。说"在服务器上部署了新版本"，指的是把新版本的程序推到机器上运行起来。搞清楚说的是硬件还是软件，很多讨论的含义才能对准。

## 一台机器扛不住时：负载均衡与多实例

一个后端服务刚上线时，可能一台机器完全够用。随着用户量增长，单台机器的 CPU、内存、网络带宽都会逐渐成为瓶颈。解决方法是横向扩展——启动多个相同的服务实例，把请求分散到这几台机器上处理，不让任何一台被打垮。

多实例本身不够，还需要解决一个问题：请求到来时，发给哪一台？**负载均衡**（Load Balancer，典型实现如 Nginx，在多个服务实例之间分发请求、避免单点过载的组件）负责做这件事。

它站在这几台服务器前面，接收所有进入的请求，根据策略决定每个请求发给哪个实例——可以简单地轮流分发（轮询），可以优先发给当前连接数最少的实例，也可以按请求的某个特征做更精细的路由。

负载均衡带来两个直接效果。流量被分散后，每个实例承受的压力降低，整体吞吐量随实例数量线性增长，需要应对更高流量时只要加机器。另一个效果是故障隔离：某个实例无法响应时，负载均衡把它从转发列表里摘掉，流量自动转向其他健康的实例，用户感知不到单台机器的故障。

## 单体服务到微服务：拆开各自管一摊

早期一个系统往往是一个整体——所有功能（用户、订单、商品、支付）的代码都在同一个仓库里，编译成同一个可执行文件，部署到同一台机器上运行。这种架构叫单体（Monolith）。

单体在系统规模小时很高效：代码在一起，函数调用直接跳转，没有网络开销，开发调试都简单。

问题出现在规模增大之后。代码量增长到几十万行时，每次改动都要重新编译部署整个应用，一个小功能的发布要把整套系统重启一遍。不同模块的代码相互依赖越来越紧，牵一发动全身，支付模块需要扩容时也只能整体扩容，没办法单独给某个模块加机器。

微服务（Microservices）的思路是把单体按业务能力拆开：用户、订单、支付各自成独立服务，拥有自己的代码仓库、数据库和发布周期，服务之间通过 API 互相调用。

拆开之后，各服务按各自的负载独立扩容——订单服务压力大时只扩容订单服务，不影响其他服务的资源分配。发布节奏也解耦了：支付团队可以独立上线新版本，不必等所有团队准备好才能整体重启。某个服务发生故障，其他服务仍然可以继续运行，故障影响范围被限制在出问题的那一块。

代价同样是真实的。原本一次函数调用变成了网络请求，延迟从微秒级变成毫秒级，还可能失败。数据分散在多个服务各自的数据库里，跨服务的查询和数据一致性变得复杂。系统从一个进程变成几十个服务，运维复杂度大幅上升。

微服务不是越拆越好，拆的粒度和时机需要根据团队规模和业务复杂度来权衡，过早引入会带来不必要的运维负担。

## 高可用：怎么让用户感觉不到故障

**高可用**（High Availability，HA，工程上通常用 SLA 衡量，99.9% 的可用性意味着一年内停机时间不超过约 8.7 小时，99.99% 不超过 52 分钟）描述的是一种目标：不能因为一台机器宕了或一个进程崩了，整个服务就停摆。

达到高可用的基本前提是消除单点。任何只有一份副本的关键组件，一旦出问题整条链路就断了，这就是**单点故障**（Single Point of Failure，SPOF）。

消除单点的方式是在每一层引入冗余：服务层挂多个实例，数据库做主从复制让从库在主库故障时接管，负载均衡本身也部署两份互为备份。哪一层没有冗余，哪一层就是潜在的致命弱点。

实例挂掉之后多快能恢复，同样关键。服务进程崩溃后，需要有进程管理工具监测到这件事，自动把进程重新拉起；如果一台机器整体故障，需要在其他机器上快速拉起新的实例来补充容量。这个过程越快，用户受到的影响窗口越短。

健康检查（Health Check）是负载均衡判断实例是否可用的手段：定期向每个实例发探测请求，实例正常响应则保持在转发列表里，探测失败达到阈值就摘掉，等实例恢复后再加回来。有了健康检查，负载均衡才能做到自动剔除故障实例，而不需要人工介入。

高可用不是单一技术点，而是多层冗余共同作用的结果：多实例分散风险、健康检查自动发现故障、快速恢复缩短影响时间。这三件事配合起来，才能让整个服务在局部故障时对外表现得像什么都没发生过。
