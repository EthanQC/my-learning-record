一个功能从"我们要做这件事"到用户能实际用上，中间要经过需求与设计、开发、Code Review、测试、灰度发布、全量发布、监控告警、回滚等环节。每一环要解决的问题不同，涉及的角色也不同，这篇把每一环分开讲清楚。

## 需求与设计：在写代码前先把问题想清楚

需求阶段要解决的问题是：这个功能到底做什么、给谁用、做到什么程度。产品经理（PM）负责这件事，产出是产品需求文档（PRD），描述目标用户是谁、要解决什么场景、每一个交互细节应该怎么走。PRD 写得不清楚，开发到一半才发现方向不对，代价远高于提前多花一天打磨需求。

技术设计要解决的问题是：用什么架构实现、数据存在哪里、接口怎么设计、哪些地方有性能风险。这部分由开发工程师（有时是架构师）主导，产出是技术方案文档。一个需要改三个服务的功能，如果各自埋头写代码，接口定义互相矛盾，合并时才发现对不上，损失的时间远超事先拉一次对齐会议。

UI 设计在这个阶段也并行展开。设计师把产品需求翻译成具体的界面稿，交互逻辑、视觉规范都在设计稿里体现，前端工程师按设计稿实现，而不是凭感觉猜界面该长什么样。

## 开发与 Code Review：写代码与互相检查

开发阶段解决的问题是把技术方案转化成可运行的代码。开发工程师在自己的功能分支上工作，前端实现界面和交互，后端实现业务逻辑和接口，不同的分支并行推进，互不影响主干的稳定。

**Code Review**（代码评审）是代码合入主干前的检查环节。提交改动的工程师创建 Pull Request（PR，向代码仓库主干提交改动的申请），团队成员作为 reviewer 阅读代码变更，提出问题和建议。

Code Review 要解决几件事。潜在的 bug——边界条件没有处理、并发场景下有数据竞争——往往在 review 时比运行时更容易被发现。设计层面的问题也能在这里浮出：一个函数承担了过多职责、某个公共模块被重复实现、命名让人无法理解意图。此外，reviewer 在 review 代码的过程中也在学习这段代码做了什么，团队成员之间的知识因此流通，不会出现某个模块只有一个人能看懂的状态。

Code Review 不是吹毛求疵，也不是走过场的橡皮章。reviewer 和提交者都对代码质量负有共同责任，review 不通过的代码不应该合入主干。

## 测试：把可能出错的地方提前找出来

测试阶段要解决的问题是：找出代码在各种输入和使用场景下可能出错的地方，在用户看到之前修掉。

**单元测试**（Unit Test）由开发工程师自己写，验证一个函数或一个独立模块的行为是否符合预期。输入是什么、期望输出是什么，都在测试用例里写清楚。单元测试覆盖的范围小但运行极快，改动某处代码后跑一遍，几秒内就能知道有没有破坏已有行为。

**集成测试**（Integration Test）验证多个模块组合起来是否正常工作，覆盖的范围包括接口调用链、数据库读写等真实依赖。场景更接近生产，但运行时间也相应更长。

**QA**（质量保障工程师，Quality Assurance）在代码集成完成后介入，从用户视角出发做更全面的功能验证，也负责编写自动化测试脚本覆盖回归场景——确保新改动没有把原本正常的功能弄坏。性能测试通常也在这个阶段进行，确认在预期流量下接口的响应时延和成功率是否达标。

测试不能保证没有 bug，它能做的是大幅降低已知类型问题逃逸到线上的概率。

## 上线：灰度发布与全量发布的存在意义

测试通过之后，代码并不是直接推给所有用户。即使测试覆盖得再全，线上真实流量的复杂程度仍然超出测试环境的模拟范围。用户的设备、网络条件、操作习惯，任何一个维度都可能触发测试没有覆盖到的路径。

**灰度发布**（Canary Release，也叫灰度测试或金丝雀发布）解决的问题正是这个：不直接全量，而是先把新版本推给一小部分用户（比如 1%、5%），观察这部分流量的错误率、响应时延、业务指标是否正常，确认没有问题再逐步扩大比例。

灰度存在的原因是：如果直接全量，一个上线后才暴露的问题会立刻影响所有用户；灰度把风险限制在一个可控的比例内，问题出现时可以快速回退，不必让所有用户都经历故障。灰度比例的扩大通常由发布系统（如 Argo Rollouts、阿里云 MSE 流量管控等工具的能力）或人工判断控制，每一步都等前一步的指标稳定后再推。

**全量发布**是灰度各阶段都通过后，把新版本推到所有服务实例的步骤。全量发布通常也是渐进的，不是一瞬间全部切换，而是逐步替换旧实例。发布完成后不代表可以立刻结束关注——新版本下的指标仍然需要持续观察。

## 上线之后：监控、告警、回滚

服务上线后要解决的问题是：系统出了问题能不能第一时间发现，发现后能不能快速定位和恢复。这三件事分别对应监控、告警、回滚。

**监控**（Monitoring）是持续采集系统运行数据的机制。典型的监控指标包括：接口的请求量（QPS）和错误率、响应时间分布（P50/P95/P99 延迟）、系统资源使用率（CPU、内存、磁盘）、业务指标（下单成功率、支付成功率）。监控数据构成对系统状态的持续可见性，没有监控就等于在黑暗里开车——系统坏掉了只能等用户反馈。

**告警**（Alerting）是在监控数据偏离正常范围时主动通知值班工程师的机制。错误率突然从 0.1% 升到 5%，接口 P99 延迟超过设定阈值，告警把这些异常从监控大盘里提炼出来，推送给需要响应的人。告警规则设置得太松，异常出现时无人知晓；设置得太严，大量误报会让工程师产生告警疲劳，真正的问题反而被淹没。告警不是监控的替代品，两者配合才能既感知全局状态、又在关键时刻主动唤人。

**回滚**（Rollback）是当新版本被确认有问题时，迅速恢复到上一个稳定版本的操作。发布系统保留了每次发布的版本记录，回滚通常是把流量切回旧版本的镜像，执行时间以分钟计。回滚不是失败，而是故障处理流程里预设的一个正常选项——在问题原因还没查清楚之前，先止血、先恢复服务，是比带着故障慢慢排查代价更小的选择。

三者的关系是：监控让问题可见，告警让人及时知道，回滚在必要时快速恢复。这条链路健全，上线后的风险才有兜底。
