前端代码的运行环境是浏览器，这决定了它与后端服务在几乎所有方面的差异：语言选择、执行时机、与用户的距离。理解前端，先要明确浏览器作为运行时承担了什么。这一切围绕 HTML/CSS/JS 三种语言、浏览器的渲染机制以及现代前端的组织方式展开。

## 浏览器是前端的运行环境

后端代码跑在服务器上，部署一次就一直运行，程序员能控制运行环境的操作系统、配置和依赖版本。前端代码则是被服务器发送到用户的浏览器里，在用户的设备上执行——程序员没办法控制用户用的是 Chrome 还是 Safari，用的是 Windows 还是手机。

浏览器不只是一个"网页阅读器"，它是一个完整的运行时环境：内置 JavaScript 引擎负责执行脚本，布局引擎负责把结构和样式转化成视觉输出，网络层负责按需拉取资源。前端代码的所有能力和限制，都由这个环境划定。

这也是为什么前端工程师需要关注"浏览器兼容性"——不同浏览器或不同版本对同一段代码的支持程度可能不同，前端的运行环境不由自己控制，而是分散在用户的各种设备上。

## HTML、CSS、JavaScript 各自的角色

一个网页由三种语言协同构成，各自负责不同层面的事情。混淆了三者的职责，会让代码在规模增大时难以维护。

HTML（HyperText Markup Language，超文本标记语言）描述的是**结构**——页面上有什么内容，以什么层级组织。一个标题、一段文字、一张图片、一个按钮，都用 HTML 标签声明出来。HTML 本身不决定这些内容长什么样，也不决定点击按钮会发生什么，它只回答"这里有什么"。

CSS（Cascading Style Sheets，层叠样式表）描述的是**样式**——内容以什么形式呈现给用户。字体大小、颜色、布局方式、间距、动画效果，都由 CSS 控制。同一份 HTML 结构，换一套 CSS 就能呈现出完全不同的视觉效果，CSS 解决的问题是"内容看起来是什么样子"。

JavaScript（JS）描述的是**行为**——页面在用户操作时如何响应。点击按钮触发提交、滚动到某个位置触发动画、输入内容时实时验证格式，这些动态交互逻辑都由 JS 实现。JS 是一门完整的编程语言，可以修改页面内容、发起网络请求、操控浏览器的各类 API。

三者分工让代码各管其职：调整布局不动 JS，修改交互逻辑不改 HTML 结构，这种分离在项目规模增大后效果尤为明显。

## 浏览器渲染的核心产物：DOM、渲染树与像素

浏览器把一个网页变成屏幕上的像素，依赖几个关键的中间产物。

**DOM**（Document Object Model，文档对象模型）是浏览器解析 HTML 后在内存中建立的树形结构。每个 HTML 标签变成树上的一个节点，标签的嵌套关系变成父子关系。DOM 是页面内容的程序化表示——JS 对页面的所有修改，本质上都是在操作这棵 DOM 树。

**渲染树**（Render Tree）是 DOM 和 CSS 信息合并后的产物。浏览器解析 CSS 得到每个节点应应用的样式规则，再将其与 DOM 合并，生成只包含需要实际显示的节点及其计算好的样式的渲染树。注释、`display: none` 的元素不会出现在渲染树里。

**像素**是渲染树经过布局和绘制后的最终产物。布局阶段计算每个节点在页面上的精确位置和尺寸，绘制阶段把内容和样式画到屏幕上。JS 修改了 DOM 或样式后，浏览器可能重新走部分布局或绘制流程——位置尺寸改变触发**回流**（Reflow），只有视觉样式改变触发**重绘**（Repaint）。

频繁触发回流会让页面变卡，是前端性能优化的常见关注点。理解到"DOM 是产物、回流有代价"这一层就够了。

## 单页应用与传统多页应用

传统的网页应用（多页应用，MPA）每次跳转页面都需要浏览器向服务器发起新的请求，服务器返回完整的 HTML，浏览器清空当前内容、重新解析和渲染整个页面。访问首页、点击"关于我们"、点进一篇文章，每步都是一次完整的页面刷新——包括导航栏、页脚这些其实没变的部分。

单页应用（Single Page Application，SPA）的做法不同：浏览器只在最开始加载一个 HTML 文件和必要的 JS 代码，后续的页面切换由 JS 代码直接修改 DOM 更新内容，数据通过 API 请求拉取后填入，不再向服务器请求新的 HTML。从用户视角看，URL 变化了、内容也切换了，但浏览器始终没有刷新。

SPA 带来的好处是交互响应更快——切换内容时不必等服务器返回整页 HTML，只拉取必要的数据，界面更新几乎即时完成，使用感更接近桌面应用。以 React 为代表的现代前端框架（React 是由 Meta 开源的、用于构建用户界面的 JS 库）就是为构建 SPA 而设计的。

代价也是真实的。初次加载需要下载完整的 JS 文件，包体积大时首屏耗时比传统多页应用更长；搜索引擎的爬虫默认不执行 JS，依赖 JS 动态生成内容的 SPA 对 SEO（Search Engine Optimization，搜索引擎优化）不友好。前端路由和状态管理的复杂度也比多页应用高。SPA 与 MPA 各有适用场景，不是替代关系。

## 组件化：把界面切成可复用的积木

当一个页面的交互逻辑足够简单，把 HTML、CSS、JS 分散在三个文件里管理是可行的。但当界面复杂度上升——同样的"用户卡片"出现在搜索结果页、关注列表页、消息页——如果每个地方都复制一份代码，改一处就需要改三处，维护成本迅速失控。

组件化（Component）的思路是把界面的一个独立单元——结构、样式、交互逻辑——封装在一起，形成一个可复用的模块。"用户卡片"成为一个组件，接受用户数据作为输入，内部自己管理显示逻辑，用到的地方直接引用这个组件，修改一处即全局生效。

**封装**是组件的核心特性之一：组件的内部实现对外部不可见，外部只通过约定好的接口（通常叫 props，即传入的参数）来定制组件的表现，内部样式不会意外影响页面其他部分。

**可复用**是组件的另一个核心特性：同一个组件在不同场景下传入不同数据，产出不同的渲染结果，不需要为每种场景单独实现一套。

组件化不只是代码复用的手段，更是一种组织复杂界面的方式。一个复杂页面可以被分解成导航栏、侧边栏、内容区、评论等组件，每个组件内部还可以继续嵌套子组件，形成组件树。这棵树的结构与 DOM 树是对应的，但它是在代码层面组织的，方便多人协作：不同团队成员可以各自负责不同的组件，互不干扰。

组件化是现代前端框架（如 React、Vue）提供的核心能力之一，也是理解现代前端工程化的起点。掌握"把界面拆成组件、组件之间通过接口通信"这个思维方式，比记住任何具体框架的 API 都更重要。
